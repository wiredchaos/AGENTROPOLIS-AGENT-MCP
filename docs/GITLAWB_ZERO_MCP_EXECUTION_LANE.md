# GitLawb Zero MCP Execution Lane

GitLawb Zero is tracked as an optional local coding execution lane for the Agentropolis MCP Agent Kit.

Public positioning:

```text
GitLawb Zero
= local coding agent lane
= your model, your machine, your rules
= operator controlled repo execution surface
```

## MCP Role

The MCP kit should treat GitLawb Zero as a tool lane candidate, not as the brain and not as unrestricted authority.

Correct placement:

```text
MCP request
  -> classify task
  -> score risk
  -> select model lane
  -> check tool authority
  -> route to GitLawb Zero if local coding is allowed
  -> generate patch candidate
  -> run validation
  -> log receipt
```

## Authority Boundary

GitLawb Zero can help with:

- repo inspection
- local patch generation
- test command planning
- package scaffolding
- documentation updates
- MCP adapter candidates
- build workflow candidates

GitLawb Zero should not bypass:

- HERMES Dispatch
- Model Council Routing
- MCP Registry authorization
- Policy Gate
- operator approval
- audit receipts

## No Single Provider Lock

This lane supports the No Claude bootstrap direction.

The MCP kit remains provider agnostic:

```text
HERMES Dispatch selects the work lane.
Model Council Routing selects the model lane.
MCP Registry selects tool access.
Policy Gate checks authority.
GitLawb Zero may execute local coding tasks.
Receipt logs preserve auditability.
```

## Minimum Receipt Fields

Every GitLawb Zero MCP action should record:

- request id
- repo name
- branch or workspace
- model lane
- tool authority grant
- files changed
- commands proposed
- commands executed
- validation result
- approval state
- rollback notes

## Core Rule

```text
Local coding is powerful.
Powerful lanes need gates.
Gates need receipts.
```
